# 平台配置说明

本文用于说明：
- 如何启动平台
- 如何配置不同账号系统（`local` / `qkit`）
- 不同部署模式（`single` / `platform` / `agent`）含义与启动方式
- 如何配置“有项目”或“空项目”启动 Agent
- 如何在其他机器启动 Agent 节点
- 如何一键发布 Agent 新版本、如何回退版本
- 生产使用要点

---

## 1. 平台启动（基础）

### 1.1 准备
- Python 3.9+
- `git` 命令可用（如要接 SVN 还需 `svn` 命令）
- 在项目根目录安装依赖：

```bash
pip install -r requirements.txt
```

### 1.2 配置 `.env`
- 可从 `.env.simple` 复制：

```bash
cp .env.simple .env
```

Windows:

```bat
copy .env.simple .env
```

### 1.3 启动平台
- Linux/macOS:

```bash
bash start.sh
```

- Windows:

```bat
start.bat
```

- 或直接：

```bash
python app.py
```

默认访问地址：`http://127.0.0.1:8002`

## 2. 账号系统配置（local / qkit）

核心开关：

```env
AUTH_BACKEND=local
```

或

```env
AUTH_BACKEND=qkit
```

### 2.1 local 账号系统（本地用户名密码）

推荐最小配置：

```env
AUTH_BACKEND=local
FLASK_SECRET_KEY=请替换为随机密钥
ADMIN_USERNAME=admin
ADMIN_PASSWORD=请设置
ADMIN_API_TOKEN=可选，供接口鉴权
ENABLE_ADMIN_SECURITY=true
AUTH_DEBUG_MODE=false
```

说明：
- `ADMIN_USERNAME/ADMIN_PASSWORD`：本地管理员账号
- `ADMIN_API_TOKEN`：用于管理接口 Header 鉴权（`X-Admin-Token`）
- `AUTH_DEBUG_MODE=true` 仅建议本地调试使用

### 2.2 qkit 账号系统

推荐最小配置：

```env
AUTH_BACKEND=qkit
FLASK_SECRET_KEY=请替换为随机密钥
QKIT_ADMIN_USERNAME=平台管理员用户名(不含邮箱后缀)
QKIT_LOCAL_HOST=你的平台访问地址(含端口)
QKIT_LOGIN_HOST=auth2.qkit.nie.netease.com
QKIT_REQUEST_TIMEOUT_SECONDS=5
QKIT_REDMINE_API_URL=http://redmineapi.nie.netease.com/api/user
ENABLE_ADMIN_SECURITY=true
```

常用可选配置：
- `QKIT_PUBLIC_BASE_URL`：反向代理场景下建议设置
- `QKIT_AUTH_CHECK_JWT_API`：自定义 JWT 校验地址
- `QKIT_LOGIN_SERVICE` / `QKIT_LOGOUT_SERVICE`：自定义登录/登出地址
- `QKIT_LOCAL_JWT_CACHE`：是否启用本地 JWT 缓存

兼容旧变量名（可用但不建议新配置继续使用）：
- `LOCAL_HOST`, `LOGIN_HOST`, `AUTH_CHECK_JWT_API`, `LOGIN_SERVICE`, `LOGOUT_SERVICE` 等

## 3. 启动模式说明（DEPLOYMENT_MODE）

```env
DEPLOYMENT_MODE=single
```

支持值：

### 3.1 `single`（默认）
- 平台 Web + 本地后台任务都启动
- 适合单机部署、开发测试

### 3.2 `platform`
- 平台只负责控制面（Web/API/任务编排）
- 不启动本地后台任务
- 实际任务下发给远端 Agent
- 适合生产“平台 + 多 Agent 节点”架构

### 3.3 `agent`
- `python app.py` 时不启动 Flask，直接进入 Agent 循环
- 一般用于同仓一体化调试
- 生产更推荐使用 `agent/` 目录独立运行

## 4. 推荐部署方式

### 4.1 单机模式（最简单）
###### 该模式下，只需要启动一个服务即可使用完整功能（缺点：后续机器性能资源不足，不方便拓展）
1. `.env` 中配置：
   - `DEPLOYMENT_MODE=single`
   - `AUTH_BACKEND=local` 或 `qkit`
2. 启动 `start.sh` / `start.bat`

### 4.2 平台 + Agent 节点（推荐生产环境）
###### 该模式下，启动平台服务的机器只接收和显示Agent机器上报的信息，Diff等逻辑实际在Agent节点机器运行，分布式部署
###### 该模式同样可以提供 Agent 节点到不同项目自行部署（代码、excel仓库在本地服务器导出），适用于保密项目
1. 平台机 `.env`：
   - `DEPLOYMENT_MODE=platform`
   - `AGENT_SHARED_SECRET=同一密钥`
2. 启动平台
3. 在各 Agent 机器部署 `agent/` 独立目录并启动（见第 6 节）

## 5. 如何配置“有项目”或“空项目”启动 Agent

Agent 关键配置在 `agent/.env`：

```env
AGENT_PROJECT_CODES=
AGENT_DEFAULT_ADMIN_USERNAME=admin
```

### 5.1 有项目代号启动
- 给 `AGENT_PROJECT_CODES` 填多个项目代号（逗号分隔）
- Agent 注册时会申请绑定这些项目（不存在时可自动创建）

### 5.2 空项目启动（推荐先上线节点再建项目）
- `AGENT_PROJECT_CODES` 留空即可
- Agent 会正常在线，但不自动创建项目
- 后续可在平台创建项目并绑定到该 Agent

### 5.3 让某用户可在平台创建并绑定到该 Agent
- 设置：

```env
AGENT_DEFAULT_ADMIN_USERNAME=某用户名
```

- 该用户在平台可创建项目并默认绑定到该 Agent（用户会被设置为项目管理员）

## 6. 其他机器启动 Agent 节点

### 6.1 分发 Agent 方式
- 可直接复制 `agent/` 目录，或用：

```bash
python agent/build_zip.py
```

生成 zip 分发。

### 6.2 远端配置 `agent/.env`
至少设置：

```env
PLATFORM_BASE_URL=http://平台地址:8002
AGENT_SHARED_SECRET=必须与平台一致
AGENT_NAME=Remote节点-001
AGENT_HOST=
AGENT_PORT=9010
```

说明：
- `AGENT_HOST` 建议留空自动探测
- `PLATFORM_BASE_URL` 不能写成本机 `127.0.0.1`（除非平台就在同一台机器）

### 6.3 启动 Agent
- Linux/macOS:

```bash
cd agent
bash start_agent.sh
```

- Windows:

```bat
cd agent
start_agent.bat
```

## 7. Agent 发布与回退

> 下面命令在**平台项目根目录**执行。

### 7.1 发布 Agent 新版本（一键发布，其他链接到平台的 Agent 节点代码会自更新）

```bash
python scripts/publish_agent_release.py
```

常用参数：
- `--version <版本号>`
- `--notes "<发布说明>"`
- `--force`
- `--allow-dirty`（允许脏工作区发布，不建议生产使用）

脚本包装：
- Windows: `scripts\publish_agent_release.bat`
- Linux/macOS: `bash scripts/publish_agent_release.sh`

### 7.2 回滚 Agent 版本

回滚到上一版：

```bash
python scripts/publish_agent_release.py --rollback --rollback-steps 1
```

回滚到指定版本：

```bash
python scripts/publish_agent_release.py --rollback --rollback-target-version <版本号>
```

独立回滚脚本（等效）：

```bash
python scripts/rollback_agent_release.py --steps 1
python scripts/rollback_agent_release.py --target-version <版本号>
```

脚本包装：
- Windows: `scripts\rollback_agent_release.bat`
- Linux/macOS: `bash scripts/rollback_agent_release.sh`

### 7.3 管理接口（管理员）
- `GET /api/agents/releases/admin/list`
- `POST /api/agents/releases/admin/rollback`

可通过登录管理员会话或 `X-Admin-Token` 调用。

## 8. Agent 自更新配置

`agent/.env`：

```env
AGENT_AUTO_UPDATE_ENABLED=true
AGENT_AUTO_UPDATE_CHECK_INTERVAL_SECONDS=300
AGENT_AUTO_UPDATE_REQUEST_TIMEOUT_SECONDS=15
AGENT_AUTO_UPDATE_DOWNLOAD_TIMEOUT_SECONDS=120
AGENT_AUTO_UPDATE_INSTALL_DEPS=true
AGENT_AUTO_UPDATE_PIP_TIMEOUT_SECONDS=900
```

行为说明：
- Agent 定时检查最新 release
- 有新版本时下载、校验 SHA256、覆盖文件、自动重启
- 如开启 `AGENT_AUTO_UPDATE_INSTALL_DEPS`，会先执行 `python -m pip install -r requirements.txt`
  - 这里的 `python` 指“当前运行 Agent 的解释器”（`sys.executable`）
  - 用 venv 启动 Agent：依赖安装到该 venv
  - 用系统 Python 启动 Agent：依赖安装到系统环境（不推荐生产）
- 因此该能力**不只依赖 venv**，两种启动方式都可触发；但强烈建议使用 venv 以避免系统环境污染和权限问题

## 9. 使用要点（建议给用户）

1. `AGENT_SHARED_SECRET` 必须在平台和所有 Agent 完全一致。  
2. 生产建议固定 `DEPLOYMENT_MODE=platform`，Agent 独立部署在业务节点。  
3. 平台和 Agent 机器时间尽量同步（NTP），便于日志、心跳和排障。  
4. 发布前尽量保持 git 工作区干净，发布后先观察一个 Agent 的升级日志再全量放开。  
5. 回滚只会切换 latest 指向，不会删除历史 release 目录。  
6. 若远端 Agent 无法升级，优先检查：
   - 平台地址可达
   - 密钥一致
   - 管理接口可访问
   - 节点磁盘空间与 Python/pip 环境  
7. 建议将平台数据库从默认 SQLite 迁移到 MySQL/PostgreSQL（生产场景）。
